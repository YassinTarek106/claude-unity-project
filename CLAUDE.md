## Project Overview

Unity 6 (6000.3.7f1) project using the **Universal Render Pipeline (URP)** v17.3.0. Currently a greenfield project with no custom scripts — built from the URP blank template with input system preconfigured.

## Key Technologies

- **Render Pipeline**: URP with separate PC and Mobile renderer assets (`Assets/Settings/`)
- **Input System**: New Input System v1.18.0 — bindings defined in `Assets/InputSystem_Actions.inputactions` with Player actions (Move, Look, Attack, Interact, Jump, Sprint, Crouch, etc.) and control schemes for Keyboard/Mouse, Gamepad, Touch, Joystick, XR
- **AI Navigation**: NavMesh package installed (com.unity.ai.navigation 2.0.9)
- **Test Framework**: com.unity.test-framework v1.6.0 installed, no tests written yet

## Architecture

- **Scenes**: `Assets/Scenes/SampleScene.unity` — single scene with Camera, Directional Light, and Global Volume
- **Rendering**: Two quality tiers configured — **PC** (higher quality, 4-weight skinning) and **Mobile** (lower quality, 2-weight skinning) with separate URP renderer assets
- **Platform targets**: Standalone (PC), Android (min SDK 25), iOS (15.0+), Web, Xbox, Switch, XR

## Unity-Specific Conventions

- **Serialization**: Force Text mode — all `.unity`, `.asset`, `.prefab` files are YAML (git-friendly)
- **Line endings**: Unix (LF)
- **Meta files**: Every asset has a `.meta` file — always commit them alongside the asset. Never delete a `.meta` without deleting its asset, and vice versa
- **Git LFS**: Enabled for binary assets
- **Scripts go in**: `Assets/` (organize by feature). Use `Assets/Editor/` for editor-only scripts
- **Tests go in**: `Assets/Tests/EditMode/` and `Assets/Tests/PlayMode/` with corresponding `.asmdef` files referencing `UnityEngine.TestRunner` and `UnityEditor.TestRunner`

## Architecture Rules

Non-negotiable. Every script must follow these.

### VContainer Dependency Injection

All services registered and injected through VContainer as interfaces.

**Banned:**
- `FindObjectOfType`
- Singletons (static instance pattern)
- Static access to services
- Service locator pattern
- `new Service()` in MonoBehaviours

### Custom Event System

Design and build your own event bus from scratch.

**Banned:**
- Unity's built-in `UnityEvent` / `UnityAction` for cross-service communication
- ScriptableObject-based event systems (SO event channels)
- Any third-party event library (UniRx, MessagePipe, etc.)

**Required:**
- Typed events (not strings)
- Subscribe / Unsubscribe with proper cleanup
- Decoupled publishers and subscribers

The design is entirely yours. Channel-based, bus-based, observer, mediator — your choice.

### Audio Service

Centralized audio through DI. No raw `AudioSource.Play()` calls in gameplay code.

- Interface-based (e.g., `IAudioService`)
- Sounds mapped by enum, not by string or direct clip reference
- Supports overlapping sounds

### UI System

Reactive, event-driven.

- UI controllers subscribe to events and update themselves
- No `SetActive(true/false)` scattered through gameplay code
- Each screen/panel is a separate controller
- Injected through VContainer

### Clean Code

- No magic numbers — use constants or config ScriptableObjects
- No God classes — no single class over ~150 lines
- Interface Segregation — small, focused interfaces
- No `#region`
- No commented-out code
- Consistent naming — follow C# and Unity conventions (PascalCase for types/methods, camelCase for locals/parameters, `_camelCase` for private fields)
